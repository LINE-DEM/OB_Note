# 🗺️ 编程语言实现地图

> _"你必须要有一张地图，无论它是多么粗糙。否则你就会到处乱逛。"_ —— J.R.R. Tolkien

## 核心比喻：翻越一座山

```
源代码                                                    可执行程序
  │                      ⛰️ 山顶                              │
  │              ┌─── 静态分析 ───┐                           │
  │         语法分析             中介码(IR)                    │
  │    词法扫描                       优化                     │
  ▼  /源文本\                             代码生成  \运行时/    ▼
[字符串] ──上山(理解语义)──► [最高抽象] ──下山(生成指令)──► [执行]

上山 = 前端（理解人写的代码）
下山 = 后端（生成机器能跑的东西）
```

## 2.1 完整流水线：八个阶段

### 阶段总览

```
┌─────────────────────────────────────────────────────────────────┐
│                        前  端  (Front End)                       │
│  ┌──────────┐    ┌──────────┐    ┌──────────────┐               │
│  │ 源代码    │───►│ 词法扫描  │───►│   语法分析    │───►           │
│  │ Source   │    │ Scanner  │    │    Parser    │               │
│  └──────────┘    └──────────┘    └──────────────┘               │
│   字符串流          Token序列          AST语法树                   │
└─────────────────────────────────┬───────────────────────────────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────┐
│                       中  端  (Middle End)                       │
│           ┌──────────────┐    ┌──────┐    ┌──────────┐          │
│      ────►│   静态分析    │───►│  IR  │───►│   优化   │───►      │
│           │Static Analy. │    │中介码│    │Optimize  │          │
│           └──────────────┘    └──────┘    └──────────┘          │
│            绑定/类型检查      平台无关表示    常量折叠等            │
└─────────────────────────────────┬───────────────────────────────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────┐
│                        后  端  (Back End)                        │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐                   │
│  │ 代码生成  │───►│  虚拟机   │───►│  运行时  │                   │
│  │ Code Gen │    │    VM    │    │ Runtime  │                   │
│  └──────────┘    └──────────┘    └──────────┘                   │
│  字节码/机器码      模拟执行        GC/类型追踪                    │
└─────────────────────────────────────────────────────────────────┘
```

---

### 阶段①：词法扫描 Scanner / Lexer

**输入** → 原始字符流 **输出** → Token（标记）序列

```
"var average = (min + max) / 2;"

         词法扫描器
              │
              ▼

[var] [average] [=] [(] [min] [+] [max] [)] [/] [2] [;]

✂️ 同时丢弃：空格、换行、注释（对语言无意义）
```

> [!NOTE] Token 类型举例
> 
> - 单字符：`(` `)` `+` `-` `;`
> - 多字符：数字 `123`、字符串 `"hi!"`、标识符 `average`

---

### 阶段②：语法分析 Parser

**输入** → Token 序列（扁平） **输出** → AST 抽象语法树（有层级）

```
[var] [average] [=] [(] [min] [+] [max] [)] [/] [2] [;]

              语法分析器
                  │
                  ▼

               var_decl
                  │
           ┌──────┴──────┐
        average          ÷
                      ┌──┴──┐
                      +     2
                   ┌──┴──┐
                  min   max

👉 反映了表达式的嵌套关系：(min+max)/2
❌ 语法出错时：报告 syntax error
```

---

### 阶段③：静态分析 Static Analysis

**做什么** → 理解"名字"的含义

```
表达式 a + b

词法/语法阶段只知道：有两个名字叫 a 和 b，做加法
静态分析才能回答：
  ┌─ a 是哪里定义的？局部变量？全局变量？
  ├─ b 的类型是什么？
  └─ a 和 b 能相加吗？（静态类型语言会在此 type check）

分析结果存哪里？
  ┌─────────────────────────────────────────┐
  │  ① AST节点的属性字段（直接附加在树上）    │
  │  ② 符号表 Symbol Table（独立查找表）      │
  │  ③ 转换为全新的数据结构（IR，见下阶段）   │
  └─────────────────────────────────────────┘
```

> [!TIP] 前端边界 扫描 + 语法分析 + 静态分析 = **前端** 之后的部分进入**中端**和**后端**

---

### 阶段④：中介码 IR（Intermediate Representation）

**为什么需要 IR？** → 解耦前端语言和后端平台

```
没有 IR 时（需要 M×N 个编译器）：

Pascal ──► x86
Pascal ──► ARM
Pascal ──► SPARC
C      ──► x86      ← 需要写 9 个！
C      ──► ARM
C      ──► SPARC
Fortran──► x86
Fortran──► ARM
Fortran──► SPARC

有了 IR（只需要 M+N）：

Pascal ──┐
C      ──┼──► IR ──┬──► x86    ← 只需写 3+3=6 个！
Fortran──┘         ├──► ARM
                   └──► SPARC
```

> [!EXAMPLE] 现实例子 GCC 就用了这个思路，前端支持 C/C++/Fortran 等，后端支持 x86/ARM/MIPS 等

---

### 阶段⑤：优化 Optimization

**原则**：语义不变，效率更高

```
常量折叠示例：

优化前：
  pennyArea = 3.14159 * (0.75 / 2) * (0.75 / 2)
                    ↓ 编译器直接算好
优化后：
  pennyArea = 0.4417860938

其他优化技术：
  • 死码删除     (Dead Code Elimination)
  • 循环展开     (Loop Unrolling)
  • 公共表达式消除 (CSE)
  • 内联展开     (Inlining)
```

> [!WARNING] 本书策略 本书大量跳过优化章节。Lua / CPython 的优化重心在**运行时**而非编译时

---

### 阶段⑥：代码生成 Code Generation

**决策点**：生成真实机器码 还是 字节码？

```
                     代码生成
                         │
            ┌────────────┴────────────┐
            ▼                         ▼
      🖥️ 原生机器码                📦 字节码(Bytecode)
      (x86 / ARM 汇编)             (虚拟指令集)
            │                         │
     ✅ 速度极快                  ✅ 跨平台可移植
     ❌ 与架构强绑定              ❌ 需要VM来执行
     ❌ 生成复杂                  ✅ 生成相对简单

字节码示例（对应 average = (min+max)/2）：
  LOAD min
  LOAD max
  ADD
  LOAD 2
  DIV
  STORE average
```

---

### 阶段⑦：虚拟机 VM

**字节码没有真实 CPU 能直接跑，怎么办？**

```
方案A：为每个目标平台写一个小编译器
  字节码 ──► 平台小编译器 ──► x86 机器码
  字节码 ──► 平台小编译器 ──► ARM 机器码
  （字节码充当 IR 角色）

方案B：写一个虚拟机 VM（本书第二个解释器的方式）
  字节码 ──► VM（用C写的程序）──► 模拟执行
  
  优点：用C实现VM，哪里有C编译器就能跑
  缺点：每条指令都要在运行时模拟，比原生慢
```

---

### 阶段⑧：运行时 Runtime

**程序跑起来之后还需要什么支持？**

```
┌────────────────────────────────────────┐
│              Runtime 提供              │
├────────────────────────────────────────┤
│  🗑️ 垃圾回收器 GC                       │
│     自动管理内存，回收不用的对象          │
│                                        │
│  🏷️ 类型信息追踪                        │
│     支持 instanceof 等运行时类型检查     │
│                                        │
│  🧵 并发支持、异常处理...               │
└────────────────────────────────────────┘

Runtime 在哪里？
  编译型语言（Go）：Runtime 直接嵌入可执行文件
  解释/VM型（Java/Python/JS）：Runtime 在 VM 里
```

## 2.2 捷径与备选路线

### 四种快捷方式对比

| 方式                 | 跳过了什么                   | 代价            | 典型例子                       |
| ------------------ | ----------------------- | ------------- | -------------------------- |
| **单遍编译器**          | 跳过 AST / IR 构建，边解析边生成代码 | 限制语言设计（需前向声明） | 早期 Pascal、C                |
| **树遍历解释器**         | 跳过代码生成，直接遍历 AST 执行      | 速度慢           | 本书第一个解释器、早期 Ruby           |
| **转译器 Transpiler** | 跳过底层代码生成，输出另一种高级语言      | 依赖目标语言工具链     | TypeScript→JS、CoffeeScript |
| **JIT 即时编译**       | 不提前编译，运行时动态编译热点为机器码     | 实现极复杂         | JVM HotSpot、V8、CLR         |

---

### 单遍编译器

```
普通编译器：
  源码 → 扫描 → 解析 → 分析 → IR → 优化 → 代码生成

单遍编译器（边解析边生成）：
  源码 → 扫描 → [解析+分析+代码生成 同时进行] → 输出

限制：看到表达式时必须已知足够信息
  → C语言需要"前向声明"就是这个原因
  → Pascal 要求类型声明必须出现在最前面
```

---

### 树遍历解释器

```
  源码 → 扫描 → 解析 → AST
                         │
                    直接遍历执行
                    （不生成任何中间代码）

适合：学习项目、脚本小工具
不适合：高性能通用语言（因为每次执行都要走树）
```

---

### 转译器 Transpiler

```
TypeScript 举例：

  .ts 源码
     │
  TS扫描器 + TS解析器（前端）
     │
  类型检查 + 分析
     │
  输出 .js 源码  ◄── 不是机器码，是另一种高级语言！
     │
  Node.js / 浏览器（用JS自己的编译管道执行）

历史演变：
  1960s → 汇编 to 汇编
  1970s → 各语言 to C（借助UNIX上到处都有C编译器）
  现在   → 各语言 to JavaScript（借助浏览器）
```

---

### JIT 即时编译

```
问题：提前编译要绑定目标架构，但不知道用户机器是什么

JIT 的解法：
  程序加载时          运行中
      │                │
  先用字节码运行   →  发现"热点"代码
                       │
                   动态编译为本机机器码
                       │
                   下次执行直接跑机器码 ⚡

例子：JVM HotSpot（名字就来自"热点"）、V8引擎
```

## 2.3 编译器 vs 解释器

### 概念辨析

```
❓ 编译器和解释器的区别？
   → 就像问"水果"和"蔬菜"的区别

           ┌─────────────────────────────────────┐
           │  编译 (Compiling) = 一种技术手段      │
           │  将源语言翻译为另一种（通常更低级）形式  │
           └─────────────────────────────────────┘

  编译器：翻译但不执行，用户自己拿去跑
          GCC 把 C 编译成机器码 → 用户运行 .exe

  解释器：拿到源码直接执行
          早期 Ruby MRI 直接遍历 AST 运行
```

---

### 维恩图：两者的关系

```
        ┌─────────────────────────────────────────┐
        │                                         │
        │   纯编译器          既编译又解释          │
        │  ┌────────┐      ┌────────────────┐     │
        │  │  GCC   │      │    CPython     │     │
        │  │  Clang │      │  Java (JVM)    │     │
        │  └────────┘      │  本书第二解释器 │     │
        │                  └────────────────┘     │
        │                        纯解释器          │
        │                      ┌─────────┐        │
        │                      │早期Ruby │        │
        │                      └─────────┘        │
        └─────────────────────────────────────────┘

CPython 是解释器，但内部有编译器（Python→字节码→VM执行）
```

---

### 三个关键问题

> [!QUESTION] 问题1：CPython 是编译器还是解释器？ **两者都是。** 用户视角是解释器（从源码直接运行），内部有编译步骤（Python→字节码），字节码在VM里执行。

> [!QUESTION] 问题2：Go 算什么？ **都算。** `go build` = 编译器；`go run` = 解释器（但内部先编译再运行）

> [!QUESTION] 问题3：编译和解释是非此即彼吗？ **不是。** 编译是"翻译技术"，解释是"执行方式"，很多实现同时用到两者。

## 总结：全局一览

```
                 ╔══════════════════════════════╗
                 ║  源代码（字符串）              ║
                 ╚══════════════╦═══════════════╝
                                │ 词法扫描
                 ╔══════════════▼═══════════════╗
     前           ║  Token 序列                  ║
     端           ╚══════════════╦═══════════════╝
                                │ 语法分析
                 ╔══════════════▼═══════════════╗
                 ║  AST 抽象语法树               ║
                 ╚══════════════╦═══════════════╝
                                │ 静态分析（绑定/类型检查）
                 ╔══════════════▼═══════════════╗
     中           ║  IR 中介码                   ║  ← 捷径：转译器在此输出高级语言
     端           ╚══════════════╦═══════════════╝
                                │ 优化
                 ╔══════════════▼═══════════════╗
     后           ║  字节码 / 机器码              ║  ← 捷径：JIT 在此动态编译
     端           ╚══════════════╦═══════════════╝
                                │ VM 执行 或 OS 直接运行
                 ╔══════════════▼═══════════════╗
                 ║  运行时（GC、类型信息...）      ║
                 ╚══════════════════════════════╝

捷径路线：
  单遍编译器   → 解析时直接输出，跳过AST/IR
  树遍历解释器 → AST后直接执行，跳过代码生成
  转译器      → 输出高级语言源码，借用其工具链
  JIT         → 运行时动态编译热点为机器码
```
