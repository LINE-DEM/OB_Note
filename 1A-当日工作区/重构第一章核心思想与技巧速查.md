# 《重构》第一章核心思想与技巧速查

## 一、为什么要重构

| 核心观点         | 说明                                                                            |
| ------------ | ----------------------------------------------------------------------------- |
| **代码是给人读的**  | 计算机是否多花了几个时钟周期来编译，又有什么关系呢？如果一个程序员花费一周时间来修改某段代码，那才要命呢——如果他理解了我的代码，这个修改原本只需一小时。 |
| **好代码的检验标准** | 人们是否能**轻而易举地修改它**                                                             |
| **何时重构**     | 当你需要添加新特性，却发现代码结构不便于修改时，先重构再添加                                                |
| **重复是万恶之源**  | 复制粘贴代码会在未来埋下逻辑不一致的隐患                                                          |

## 二、重构的核心原则

### 1. 小步前进

> *"小的步子可以更快前进，保持代码永远处于可工作状态"*

- 每次只做一个微小的修改
- **每次修改后立即运行测试**
- 频繁提交到版本控制系统
- 出错时只需回退一小步

### 2. 测试先行

> *"重构前，先检查自己是否有一套可靠的测试集"*

- 测试必须能**自我检验**（自动判断通过/失败）
- 测试是你的"bug 检测器"

### 3. 先重构再优化性能

- 大多数性能问题可以忽略
- 结构良好的代码更容易进行性能调优
- 不要过早优化

### 4.重要方向
消除重复代码。代码量减少并不会使系统运行更快，因为这对程序的资源占用几乎没有任何明显影响。然而代码量减少将使未来可能的程序修改动作容易得多。
## 三、重构的演进路径

```
长函数 → 提炼函数(分解关注点) → 移除临时变量(简化作用域) → 拆分阶段(计算与展示分离) → 引入多态(消除条件分支)
```

## 五、黄金法则速记

1. **先测试，后重构**
2. **小步修改 → 测试 → 提交 → 重复**
3. **临时变量 = 麻烦**，尽量用查询函数替代
4. **函数返回值统一命名为 result**
5. **参数名带上类型**（如 aPerformance）
6. **营地法则**：离开时代码比来时更健康

## 六、编码风格建议

- 函数返回值命名为 `result`，一眼看出它的作用
- 使用动态类型语言时，参数名带上类型前缀（如 `aPerformance`）
- 好的命名比注释更重要——代码应该自解释
- 每次重构后立即编译、测试、提交

## 七、关于性能的观点

> "大多数时候重复的循环对性能的影响都可忽略不计"

- 不要过早优化
- 先完成重构，再做性能优化
- 结构良好的代码更容易进行性能调优
- 最后得到的是既整洁又高效的代码
