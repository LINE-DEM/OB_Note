## 1. 隐式状态机

  

### 一句话定义

  

> 系统实际存在多个互斥的运行阶段，但没有用显式的状态枚举/状态机来建模，

> 而是用 **散落在不同类里的多个 bool / enum 的组合** 来隐式表达。

  

### 你的代码中长什么样

  

```

MainUIForm.mainState        (QiPaoMain / HuiFu)    ← 文件 A

MainUIForm.isAutoDaoLan     (true / false)          ← 文件 A

NetWorkGuide.isMoving       (true / false)          ← 文件 B

NetWorkGuide.isTrueMoving   (true / false)          ← 文件 B

GuideMgr.isOutingChargePoint(true / false)          ← 文件 C

Record._isRecording         (true / false)          ← 文件 D

GuideUIForm 是否打开         (open / closed)         ← 文件 E

```

  

7 个布尔/枚举 × 5 个文件 = 理论上 2^6 × 2 = **128 种组合**。

  

但实际合法的组合只有大约 5~6 种：

  

```

Idle     = QiPaoMain + !isMoving + !isRecording + GuideUI关闭

Talking  = HuiFu     + !isMoving + ...

Guiding  = QiPaoMain + GuideUI打开 + ...

Moving   = QiPaoMain + isMoving + isTrueMoving + ...

Recording= HuiFu     + isRecording + ...

```

  

**128 种组合中，120+ 种是非法状态，但代码中没有任何机制阻止它们发生。**

  

### 为什么有害

  

```

                合法状态         非法状态

                ┌──────┐      ┌──────────────────────────────┐

bool 组合空间:  │ 5~6  │      │         ~120 种               │

                │ 种   │      │  没有任何代码防止进入          │

                └──────┘      │  一旦进入就是诡异 bug          │

                              └──────────────────────────────┘

```

  

- **Bug 藏在非法组合里**：比如 `isMoving=true` 同时 `mainState=HuiFu`，这应该不存在，但没人阻止它发生

- **加新状态是噩梦**：你想加一个"导览介绍中"的状态，需要去 5 个文件里各加一个 bool，然后在 13 个文件里加 if 判断

- **不可能写出完整的测试**：你无法枚举 128 种组合逐一验证

  

### 识别口诀

  

> **看到 2 个以上 bool 字段经常被 `&&` `||` 组合判断，而且散布在不同的类里

> → 这就是隐式状态机。**

  

### 更具体的信号

  

```csharp

// 信号 1：多个 bool 联合判断

if (mainState == HuiFu && !isMoving && Record.Instance._isRecording) { ... }

  

// 信号 2：isXxx + isTrueXxx 成对出现

bool isMoving;       // 语义模糊

bool isTrueMoving;   // 名字里带 "true" = 上一个 bool 不可信

  

// 信号 3：不同文件里的 bool 互相"偷看"

// 文件 A 读文件 B 的 bool，文件 B 读文件 C 的 bool

NetWorkGuide.Instance.isTrueMoving   // A 偷看 B

GuideMgr.Instance.isOutingChargePoint // A 偷看 C

```

  

### 正解：显式状态机（State Pattern 的简化版）

  

```

                    ┌─────────────────┐

                    │  AppState 枚举   │  ← 唯一的真相来源

                    │  Idle           │

                    │  Talking        │

                    │  Guiding        │

                    │  Moving         │

                    └────────┬────────┘

                             │

          TransitionTo() ────┤──── 唯一写入口

                             │

                     OnStateChanged ──→ 广播给所有关心的人

```

  

- 合法状态 = 枚举值的个数，**非法组合在类型层面就不可能存在**

- 加新状态 = 枚举加一个值 + 在 TransitionTo 的 switch 里加一个 case

- Debug = 在 TransitionTo 加断点，一切状态变更尽收眼底

  

### 快速识别清单

  

- [ ] 有 3 个以上 bool 字段影响同一个行为的分支

- [ ] 这些 bool 分布在 2 个以上的类/单例里

- [ ] 存在 `isXxx` 和 `isReallyXxx` / `isTrueXxx` 成对出现

- [ ] 加一个新"模式"需要改动超过 3 个文件

  

**命中 2 条以上 → 你需要一个显式状态机。**

## 2. 单锁多因

  

### 一句话定义

  

> 用一个 bool 来仲裁多个独立原因的"请求"，

> 导致任何一个原因的 Release 会错误地覆盖其他原因的 Lock。

  

### 你的代码中长什么样

  

```csharp

// MainUIForm.cs 第 49 行

private bool _isLanguageChangeHidden = false;

```

  

5 个不同的原因要隐藏 LanguageChange：

  

```

原因 A: 进入问答页    →  _isLanguageChangeHidden = true

原因 B: 导览介绍中    →  _isLanguageChangeHidden = true

原因 C: 气泡隐藏      →  _isLanguageChangeHidden = true

原因 D: 自动导览      →  _isLanguageChangeHidden = true

原因 E: 移动中        →  _isLanguageChangeHidden = true

```

  

问题：当原因 B 结束时（导览关闭），`GuideUIForm.Hiding()` 调用 `ShowLanguageChange()`：

  

```csharp

// GuideUIForm.cs 第 266 行

mainUIForm.ShowLanguageChange();  // → _isLanguageChangeHidden = false

```

  

**但此时原因 E（移动中）可能仍然成立！一个 bool 无法记住"还有其他人也想隐藏"。**

  

### 图解

  

```

时间线 →

  

原因 B (导览):  ████████████████░░░░░░░░░░░░░░░░

原因 E (移动):  ░░░░░░████████████████████░░░░░░░

  

_isLanguageChangeHidden:

                true─────────────false────────────

                                  ↑

                          原因B结束，调用Show

                          但原因E还在！应该继续隐藏！

  

结果：移动中语言按钮闪现 → 下一帧 Update 守卫再次 Hide → 闪烁

```

  

### 为什么有害

  

- **状态闪烁**：UI 在一帧内 Show 了又 Hide，产生视觉闪烁

- **竞态条件**：不同系统在不同时机调用 Show/Hide，结果取决于调用顺序

- **防御性守卫泛滥**：为了"兜底"，你不得不在 Update 里加每帧检查（见下一节）

  

### 识别口诀

  

> **一个 bool 被 2 个以上独立的系统写入 → 这就是单锁多因。**

  

### 更具体的信号

  

```csharp

// 信号 1：同一个 bool 被多个不相关的方法设置

public void HideLanguageChange()  { _isHidden = true;  }   // A 调用

public void FadeOutXxx()          { _isHidden = true;  }   // B 调用

// 两个方法做的事情几乎一样，但被不同系统调用

  

// 信号 2：Show 方法里有"额外条件检查"来防御其他系统

public void ShowLanguageChange(bool isOutingChargePoint = false)

{

    if(!isOutingChargePoint)

    {

        if(NetWorkGuide.Instance.isTrueMoving) return;  // ← 防御！

    }

    _isHidden = false;

}

// Show 需要知道"移动系统"的存在 = 它已经意识到有别人也在控制这个 bool

  

// 信号 3：方法签名上出现 bool 参数来"绕过"正常逻辑

ShowLanguageChange(true);   // true = "我知道正常逻辑会阻止，但我硬要 Show"

```

  

### 正解 A：引用计数 / Flags

  

```csharp

[Flags] enum HideReason { InQA=1, Guide=2, Moving=4, AutoGuide=8 }

HideReason _reasons = HideReason.None;

  

void RequestHide(HideReason r)  { _reasons |= r;  Refresh(); }

void ReleaseHide(HideReason r)  { _reasons &= ~r; Refresh(); }

void Refresh() { LanguageChange.SetActive(_reasons == HideReason.None); }

```

  

每个原因只管自己的标记位，互不干扰。只有所有原因都 Release 了才 Show。

  

### 正解 B：集中式状态机（更彻底）

  

把"谁该显示谁该隐藏"的决策权从 5 个调用方收回到一个 switch-case 里：

  

```csharp

void OnStateChanged(AppState old, AppState current)

{

    switch (current)

    {

        case Idle:    LanguageChange.SetActive(true);  break;

        case Talking: LanguageChange.SetActive(false); break;

        case Guiding: LanguageChange.SetActive(false); break;

        case Moving:  LanguageChange.SetActive(false); break;

    }

}

```

  

不再有多个系统互相竞争同一个 bool。

  

### 快速识别清单

  

- [ ] 一个 bool/字段 被 3 个以上的方法修改

- [ ] 这些方法分属不同的调用来源/系统

- [ ] Show 方法里有 `if (某个其他系统的条件) return;` 的防御代码

- [ ] 方法签名有 bool 参数用来"绕过"常规检查

  

**命中 2 条以上 → 你遇到了单锁多因。**

## 3. 轮询耦合

  

### 一句话定义

  

> 系统 B 不是被通知"状态变了"，而是每帧/每次调用时主动去读系统 A 的字段，

> 形成从 B 到 A 的隐式依赖。

  

### 你的代码中长什么样

  

13 个文件，21 个位置，全部是这个模式：

  

```csharp

// BubbleManager.cs

MainUIForm mainUIForm = UIManager.GetInstance().GetUIForm<MainUIForm>(ProConst.MAIN_UIFORM);

if (mainUIForm.mainState == MainUIForm.MainState.HuiFu)

{

    // ...做某事

}

```

  

```csharp

// Record.cs

if(mainUIForm.mainState == MainUIForm.MainState.HuiFu)

  

// XunFeiManager.cs

if (UIManager.GetInstance().GetUIForm<MainUIForm>(...).mainState == MainUIForm.MainState.QiPaoMain)

  

// InputMonitor.cs / InputMonitor1.cs / InputMonitor2.cs

if( mainUIForm.mainState == MainUIForm.MainState.QiPaoMain)

  

// AudioMgr.cs

if(mainUIForm.mainState == MainUIForm.MainState.HuiFu)

  

// ...还有 7 个文件

```

  

### 依赖关系图

  

```

         mainState (public 字段)

              │

    ┌─────────┼─────────┬──────────┬──────────┐

    │         │         │          │          │

    ▼         ▼         ▼          ▼          ▼

 Bubble    Record   XunFei     Audio     InputMon

 Manager            Manager    Mgr       ×3

    │

    ├── 还要读 NetWorkGuide.isMoving

    ├── 还要读 GuideMgr.isOutingChargePoint

    └── 还要读 Record._isRecording

  

每个箭头 = 一条隐式依赖，代码里看不到 using/import，只在运行时存在

```

  

### 为什么有害

  

**1. 霰弹式修改（Shotgun Surgery）**

  

你想把 `MainState.HuiFu` 改名为 `MainState.Conversation`？要改 13 个文件 21 处。

  

你想新增一个 `MainState.Guiding`？要去 13 个文件里逐个检查每个 if-else 是否需要加分支。

  

**2. 时序不确定性**

  

```

同一帧内的执行顺序：

  

Update() 执行顺序（由 Unity 决定，不可控）：

  1. BubbleManager.Update()  → 读到 mainState = QiPaoMain

  2. MainUIForm.Update()     → 把 mainState 改成 HuiFu

  3. AudioMgr.SomeCallback() → 读到 mainState = HuiFu

  

同一帧内，BubbleManager 和 AudioMgr 读到了不同的值！

```

  

**3. 不可测试**

  

要测试 BubbleManager 的行为，你必须构造一个完整的 UIManager + MainUIForm +

NetWorkGuide + GuideMgr + Record 环境。因为 BubbleManager 在运行时直接伸手进这些单例读状态。

  

### 识别口诀

  

> **看到 `SomeManager.Instance.someField` 被多个不相关的类直接读取

> → 这就是轮询耦合。**

  

### 更具体的信号

  

```csharp

// 信号 1：在 Update / 协程中反复读取外部状态

void Update() {

    var xxx = SomeSingleton.Instance.someState;  // 每帧都读

    if (xxx == ...) { ... }

}

  

// 信号 2：GetInstance().GetXxx().someField 链式访问（迪米特法则违反）

UIManager.GetInstance().GetUIForm<MainUIForm>(ProConst.MAIN_UIFORM).mainState

//  ↑ 3 层穿透：UIManager → UIForm容器 → MainUIForm的内部字段

  

// 信号 3：同一个字段在 grep 中出现在 5+ 个不同文件

// $ grep -r "mainState" --include="*.cs" | wc -l

// → 25+  ← 危险信号

```

  

### 正解：观察者模式（Observer / Event）

  

```

         AppStateManager

              │

        TransitionTo()

              │

        OnStateChanged ──广播──┬──────────┬──────────┐

              │                │          │          │

              ▼                ▼          ▼          ▼

         MainUIForm      BubbleManager  AudioMgr  Record

         (订阅)          (订阅)         (订阅)    (订阅)

```

  

- **推 vs 拉**：状态变了主动推送，而非每帧被动拉取

- **解耦**：订阅者不需要知道状态存在哪个类里

- **时序确定**：所有订阅者在 TransitionTo 的同一调用栈中响应，不存在同一帧读到不同值

- **可测试**：Mock 一个事件源即可，不需要构造整个单例网络

  

### 快速识别清单

  

- [ ] 同一个字段被 5+ 个不同类直接读取

- [ ] 读取时需要 2 层以上的链式调用 (`A.Instance.GetB().field`)

- [ ] 字段是 public 的，且不是只读

- [ ] 修改这个字段的名字或类型需要改 5+ 个文件

  

**命中 2 条以上 → 你应该用事件/回调替代轮询。**

## 4. 守卫补丁

  

### 一句话定义

  

> 在 Update 中用每帧轮询来"兜底"修复其他系统的逻辑缺陷，

> 而不是去修复真正的问题根源。

  

### 你的代码中长什么样

  

```csharp

// MainUIForm.cs 第 233-241 行

private void Update()

{

    // ...其他逻辑...

  

    // --- 移动中强制隐藏 LanguageChange 守卫 ---

    if(NetWorkGuide.Instance.isTrueMoving)

    {

        if(GuideMgr.Instance.isOutingChargePoint)

        {

            return ;

        }

        HideLanguageChange();  // 每帧调用！

    }

}

```

  

**这段代码的潜台词是**：

> "我知道有些系统会在不该 Show 的时候 Show 出来，我管不了它们，

> 所以我在 Update 里每帧检查一遍，发现不对就强行 Hide 回去。"

  

这就是**守卫补丁** — 不去修复 ShowLanguageChange 的调用时机问题，而是用每帧轮询来掩盖。

  

### 因果链

  

```

根因：单锁多因（第 2 节）

  ↓

症状：GuideUIForm.Hiding() 在移动中错误地 Show 了 LanguageChange

  ↓

补丁：在 Update 里每帧强行 Hide

  ↓

新症状：UI 每帧 Show→Hide→Show→Hide（闪烁），性能浪费

  ↓

新补丁：HideLanguageChange() 加 early return `if (_isHidden) return;`

  ↓

掩盖了 bug，但根因未解决...

```

  

### 为什么有害

  

- **掩盖根因**：真正的 bug（谁在错误时机 Show 了）被守卫掩盖，永远不会被发现

- **性能浪费**：每帧做一次本不需要的检查和操作

- **脆弱性**：守卫本身也有 bug（比如 `isOutingChargePoint` 的 return 会跳过 Update 后续的所有逻辑）

- **传染性**：其他开发者看到这个模式后，也会在自己的 Update 里加守卫，而不是去修复根因

  

### 识别口诀

  

> **Update 里有一段逻辑，注释写着"强制"、"守卫"、"兜底"、"确保"

> → 这就是守卫补丁。它在告诉你别的地方有未修复的 bug。**

  

### 更具体的信号

  

```csharp

// 信号 1：Update 中的条件分支在"纠正"其他系统的输出

void Update() {

    if (某个外部条件) {

        强制设置某个状态();    // ← 这个状态本应由事件驱动正确设置

    }

}

  

// 信号 2：注释里有以下关键词

// "防止"、"避免"、"强制"、"兜底"、"确保"、"修复xxx的问题"

  

// 信号 3：删掉这段 Update 代码后，某些场景下 UI 会"偶尔"出错

// "偶尔" = 取决于其他系统的调用时序 = 根因是竞态条件

```

  

### 正解：修复根因

  

守卫补丁总是在暗示另一个更深层的问题。解法不是加更多守卫，而是：

  

1. **找到谁在错误时机调用了** → 修复那个调用

2. **如果调用时机正确但逻辑不够** → 用 Flags / 状态机替代 bool（见第 1、2 节）

3. **修复后守卫补丁可以安全删除** → 如果删了守卫一切正常，说明根因已经解决

  

### 快速识别清单

  

- [ ] Update 中有一段逻辑，它的唯一作用是"纠正"其他代码的输出

- [ ] 删除这段逻辑后，只在特定操作顺序下才会出 bug

- [ ] 注释中有"强制"、"兜底"、"确保"等词

- [ ] 这段逻辑在做的事情，在别的方法里也在做（重复调用）

  

**命中 2 条以上 → 不要加守卫，去修根因。**

## 5. Feature Envy（特性依恋）

  

### 一句话定义

  

> 一个类的方法频繁地访问另一个类的内部数据来做决策，

> 说明这个决策逻辑放错了位置。

  

### 你的代码中长什么样

  

```csharp

// BubbleManager.cs — 4 次伸手进 MainUIForm 读状态

MainUIForm mainUI = UIManager.GetInstance().GetUIForm<MainUIForm>(ProConst.MAIN_UIFORM);

isInHuiFuPanel = mainUI.mainState == MainUIForm.MainState.HuiFu;  // 第 1 次

  

if (mainUI != null && mainUI.mainState == MainUIForm.MainState.HuiFu)  // 第 2 次

{

    // 气泡渲染逻辑 A

}

  

if(mainUIForm.mainState == MainUIForm.MainState.QiPaoMain)  // 第 3 次

{

    // 气泡渲染逻辑 B

}

  

if (mainUI != null && mainUI.mainState == MainUIForm.MainState.HuiFu)  // 第 4 次

{

    // 气泡渲染逻辑 C

}

```

  

BubbleManager 对 MainUIForm 的内部状态的了解程度，比 MainUIForm 自己还深。

  

**它甚至需要知道枚举值的名字**（`MainUIForm.MainState.HuiFu`），这意味着 MainUIForm 的内部枚举改名 → BubbleManager 编译失败。

  

### 与轮询耦合的关系

  

```

Feature Envy 是"设计层面"的问题 — 逻辑放错了地方

轮询耦合是"实现层面"的问题 — 用轮询而非事件来获取数据

  

它们经常同时出现，互相加重：

  

  Feature Envy（逻辑放错地方）

       ↓

  需要读别人的状态

       ↓

  轮询耦合（每帧去偷看）

       ↓

  加更多的 if 判断

       ↓

  更深的 Feature Envy...

```

  

### 识别口诀

  

> **一个方法里，访问外部对象字段的次数 > 访问自身字段的次数

> → 这个方法可能放错了类。**

  

### 正解：Tell, Don't Ask（告知，别询问）

  

**反模式（Ask）**— BubbleManager 去问 MainUIForm "你现在什么状态？"

  

```csharp

// BubbleManager 主动问

var state = mainUIForm.mainState;

if (state == HuiFu) { DoA(); }

else { DoB(); }

```

  

**正确模式（Tell）**— MainUIForm（或状态机）告诉 BubbleManager "你该做什么了"

  

```csharp

// AppStateManager 广播状态变更

// BubbleManager 被告知

void OnStateChanged(AppState old, AppState current)

{

    if (current == AppState.Talking) DoA();

    else DoB();

}

```

  

区别在于：**BubbleManager 不再需要知道 MainUIForm 的存在**。它只需要知道"应用状态变了"。

  

### 快速识别清单

  

- [ ] 一个方法中 `其他对象.字段` 出现了 3 次以上

- [ ] 需要 `using` / 引用另一个类的内部枚举才能编译

- [ ] 修改被访问类的字段名 → 调用方也要改

- [ ] 同样的 `if (xxx.state == yyy)` 判断在 3+ 个不同文件中重复出现

  

**命中 2 条以上 → 考虑用 Tell Don't Ask + Observer 重构。**

## 总结：5 个坏味道的因果关系

  

```

                 ┌──────────────────┐

                 │ 1. 隐式状态机     │  ← 根因

                 │ （缺少显式建模）   │

                 └────────┬─────────┘

                          │

              ┌───────────┼───────────┐

              ▼                       ▼

   ┌──────────────────┐   ┌──────────────────┐

   │ 2. 单锁多因       │   │ 3. 轮询耦合       │

   │ (bool 仲裁冲突)   │   │ (13个文件偷看)    │

   └────────┬─────────┘   └────────┬─────────┘

            │                      │

            ▼                      ▼

   ┌──────────────────┐   ┌──────────────────┐

   │ 4. 守卫补丁       │   │ 5. Feature Envy  │

   │ (Update每帧兜底)  │   │ (逻辑放错位置)   │

   └──────────────────┘   └──────────────────┘

```

  

**修复顺序**：自上而下。修好第 1 层（引入状态机），第 2~5 层会自然消解。

  

这就是为什么重构指南中推荐 **AppStateManager 方案**而非逐个修补 —— 它治的是根因，不是症状。

## 速查表：一眼识别

  

| 坏味道 | 关键信号 | 修复模式 |

|--------|---------|---------|

| 隐式状态机 | 3+ 个 bool 联合判断；`isTrueXxx` 出现 | 显式枚举 + State Machine |

| 单锁多因 | 1 个 bool 被 3+ 个系统写入 | Flags 引用计数 / 集中决策 |

| 轮询耦合 | 同一字段被 5+ 个文件 `Instance.xxx` 读取 | Observer 事件驱动 |

| 守卫补丁 | Update 中注释写"强制""兜底" | 修复根因后删除守卫 |

| Feature Envy | 方法内 `别人.字段` > `自己.字段` | Tell Don't Ask + Observer |
